# v1.2 Step 0a — Pulp Microkernel Validation Result

**Date:** 2026-04-30
**Hardware:** Apple Silicon (NEON, 128-bit f32x4 SIMD)
**Probe:** `examples/probe_pulp_dot.rs`
**Pulp version:** 0.22.2 (transitive through faer 0.24)

## Result: Plan A is viable

At d=1536, pulp with 4 manually-unrolled accumulators matches hand-written
`std::arch` NEON within 2.7%.

| d | scalar | pulp (naive, 1 acc) | pulp ×4 (4 acc) | std::arch NEON | faer matmul |
|---|---|---|---|---|---|
| 128 | 48.60 ns | 11.91 ns | 6.72 ns | 6.73 ns | 6.76 ns |
| 384 | 119.68 ns | 40.92 ns | 15.88 ns | 15.76 ns | 16.98 ns |
| 1536 | 673.26 ns | 231.42 ns | **68.54 ns** | **66.73 ns** | 70.55 ns |

**pulp ×4 / NEON ratios:** 0.998 (d=128), 1.008 (d=384), 1.027 (d=1536). All within 15%.

## Critical caveat: naive pulp is 3.47× slower

| d | pulp (naive) / NEON |
|---|---|
| 128 | 1.77× slower |
| 384 | 2.60× slower |
| 1536 | 3.47× slower |

A pulp dot product written as a single-accumulator loop is bottlenecked
on FMA latency, exactly like a single-accumulator `std::arch` impl would
be. **Pulp does not free you from manual unrolling.** Every microkernel
must use multiple accumulators (4 on NEON for f32) to hit the FMA
throughput ceiling.

This is a finding that should shape the v1.2 plan:
- The "pulp gives us per-architecture variants automatically" claim is
  true for the *primitives*, not for the *kernel-level optimization*.
- Per-arch tuning (Milestone 8 in the execution plan) might still be
  needed if the optimal accumulator count differs across architectures
  (e.g. AVX-512 might want 8 accumulators because of higher FMA latency
  or more registers).
- Step 0a should be re-run on AVX2 hardware before milestone 4 to
  confirm pulp ×N matches AVX2+FMA there too.

## Pulp 0.22 API notes for the implementation

Method names follow `<op>_f32s` convention (not `f32s_<op>`):

- `simd.splat_f32s(value)` — broadcast scalar to vector.
- `simd.mul_add_e_f32s(a, b, c)` — fused `a*b + c` (uses hardware FMA).
- `simd.reduce_sum_f32s(v)` — horizontal sum to scalar.
- `S::as_simd_f32s(slice) -> (&[Self::f32s], &[f32])` — split slice into
  vector chunks + scalar tail. Associated function on the trait.

There is **no `add_f32s`** for plain vector add. To combine multiple
accumulators, either:
- Reduce each accumulator separately and add scalars (what `pulp x4` does
  in the probe), or
- Use `mul_add_f32s(acc1, splat(1.0), acc0)` — costs an extra FMA.

The "reduce separately" approach is fine for end-of-loop combining since
it runs once per call, not per iteration.

## Decision

**Proceed with Plan A** from `docs/v1.2-execution-plan.md`:
- Pulp + fused kernel, ~7-9 weeks.
- Every kernel hand-unrolled to N accumulators, where N depends on arch:
  - NEON (Apple Silicon): 4 accumulators.
  - AVX2: probably 4-8 accumulators (re-validate when CI runs).
  - AVX-512: probably 8 accumulators (manual validation, no CI).

   `fused_threshold_emit_tile` using faer matmul as the inner compute,

   Wait until PR #2 is review-stable.
2. **Cross-arch sanity** (optional but cheap): re-run this probe on a
   Linux x86 machine with AVX2 to confirm pulp ×4 (or ×8) matches
   `std::arch` AVX2+FMA there. Could be done via GitHub Actions.

The v1.2 plan's Step 0 decision criteria expected pulp's microkernel
structure to be the variable. The actual variable turned out to be
*how the kernel is structured* — a naive impl is below threshold, an
unrolled impl is on target. The plan is correct; it just needs an
explicit "every kernel is multi-accumulator" requirement.
