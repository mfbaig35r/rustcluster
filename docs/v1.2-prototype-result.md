- **Step 0b** (further down): fused-architecture spike with faer inner.

# v1.2 Step 0a — Pulp Microkernel Validation Result

**Date:** 2026-04-30
**Hardware:** Apple Silicon (NEON, 128-bit f32x4 SIMD)
**Probe:** `examples/probe_pulp_dot.rs`
**Pulp version:** 0.22.2 (transitive through faer 0.24)

## Result: Plan A is viable

At d=1536, pulp with 4 manually-unrolled accumulators matches hand-written
`std::arch` NEON within 2.7%.

| d | scalar | pulp (naive, 1 acc) | pulp ×4 (4 acc) | std::arch NEON | faer matmul |
|---|---|---|---|---|---|
| 128 | 48.60 ns | 11.91 ns | 6.72 ns | 6.73 ns | 6.76 ns |
| 384 | 119.68 ns | 40.92 ns | 15.88 ns | 15.76 ns | 16.98 ns |
| 1536 | 673.26 ns | 231.42 ns | **68.54 ns** | **66.73 ns** | 70.55 ns |

**pulp ×4 / NEON ratios:** 0.998 (d=128), 1.008 (d=384), 1.027 (d=1536). All within 15%.

## Critical caveat: naive pulp is 3.47× slower

| d | pulp (naive) / NEON |
|---|---|
| 128 | 1.77× slower |
| 384 | 2.60× slower |
| 1536 | 3.47× slower |

A pulp dot product written as a single-accumulator loop is bottlenecked
on FMA latency, exactly like a single-accumulator `std::arch` impl would
be. **Pulp does not free you from manual unrolling.** Every microkernel
must use multiple accumulators (4 on NEON for f32) to hit the FMA
throughput ceiling.

This is a finding that should shape the v1.2 plan:
- The "pulp gives us per-architecture variants automatically" claim is
  true for the *primitives*, not for the *kernel-level optimization*.
- Per-arch tuning (Milestone 8 in the execution plan) might still be
  needed if the optimal accumulator count differs across architectures
  (e.g. AVX-512 might want 8 accumulators because of higher FMA latency
  or more registers).
- Step 0a should be re-run on AVX2 hardware before milestone 4 to
  confirm pulp ×N matches AVX2+FMA there too.

## Pulp 0.22 API notes for the implementation

Method names follow `<op>_f32s` convention (not `f32s_<op>`):

- `simd.splat_f32s(value)` — broadcast scalar to vector.
- `simd.mul_add_e_f32s(a, b, c)` — fused `a*b + c` (uses hardware FMA).
- `simd.reduce_sum_f32s(v)` — horizontal sum to scalar.
- `S::as_simd_f32s(slice) -> (&[Self::f32s], &[f32])` — split slice into
  vector chunks + scalar tail. Associated function on the trait.

There is **no `add_f32s`** for plain vector add. To combine multiple
accumulators, either:
- Reduce each accumulator separately and add scalars (what `pulp x4` does
  in the probe), or
- Use `mul_add_f32s(acc1, splat(1.0), acc0)` — costs an extra FMA.

The "reduce separately" approach is fine for end-of-loop combining since
it runs once per call, not per iteration.

## Step 0a decision

**Proceed with Plan A** from `docs/v1.2-execution-plan.md`:
- Pulp + fused kernel, ~7-9 weeks.
- Every kernel hand-unrolled to N accumulators, where N depends on arch:
  - NEON (Apple Silicon): 4 accumulators.
  - AVX2: probably 4-8 accumulators (re-validate when CI runs).
  - AVX-512: probably 8 accumulators (manual validation, no CI).

The v1.2 plan's Step 0 decision criteria expected pulp's microkernel
structure to be the variable. The actual variable turned out to be
*how the kernel is structured* — a naive impl is below threshold, an
unrolled impl is on target. The plan is correct; it just needs an
explicit "every kernel is multi-accumulator" requirement.

# v1.2 Step 0b — Fused-Architecture Spike Result

**Date:** 2026-05-01
**Hardware:** Apple Silicon
**Probe:** `examples/probe_fused_spike.rs`
**Workload:** n=20000, d=1536, threshold=0.5, normalized random vectors

## Result: architectural fusion alone is **insufficient**

Sub-tile row-band architecture with faer inner compute is *slower* than
the production full-tile path at every sub-tile size we tested.

| variant | best (ms) | vs baseline |
|---|---|---|
| baseline (full 384×384 tile materialized) | 672.6 | 1.00× |
| spike sub=8 | 1736.7 | 0.39× (−61%) |
| spike sub=16 | 1173.6 | 0.57× (−43%) |
| spike sub=32 | 906.6 | 0.74× (−26%) |
| spike sub=64 | 796.0 | 0.84× (−16%) |
| spike sub=128 | 742.7 | 0.91× (−9%) |

Parity check passed for every variant: identical edge sets to baseline
(threshold=0.5 on random unit-norm d=1536 vectors yields 0 edges, which
is expected and consistent with the existing `benches/index.rs`
threshold).

## Why fusion-with-faer-inner doesn't help

1. **Faer GEMM is already cache-blocked internally.** Shrinking the call
   shape from 384×384 down to 64×384 or 8×384 drops *below* faer's
   internal sweet-spot block size. Faer pays per-call setup overhead,
   and we re-pay it many times per outer tile.
2. **The score-tile read isn't the bottleneck on Apple Silicon.** The
   baseline's 384×384 score tile is 590 KB — fits comfortably in L2
   (12+ MB on M-series). The "L1-fitting sub-tile" advantage is
   theoretical here; the actual cost is dominated by the matmul itself,
   which faer already optimizes.
3. **True fusion requires the threshold check inside the FMA loop.**
   You can't get the architectural win by reorganizing GEMM calls
   externally — you have to inline the compare-and-emit into the
   register-blocked accumulator loop. Faer doesn't expose that.

## Implication for the plan

**Plan C / C+ are off the table.** The microkernel work is
*load-bearing*, not bonus. The decision space simplifies:

| Pulp x4 from Step 0a | Action |
|---|---|
| Within 15% of std::arch (✓ — measured) | **Plan A** — pulp microkernel + fused threshold-emit kernel |
| If pulp had failed | Plan B — std::arch microkernel + fused kernel |

The fused kernel must be written as a custom register-blocked dot loop
with the threshold check fused into the inner FMA accumulation.
Faer-inner-compute experiments are a dead end.

## What this means for milestone sequencing

The v1.2 execution plan's milestone 5 ("wire batched_dot_tile into
ip_batch") and milestone 6 ("fused_threshold_emit_tile proper") stay
exactly as written. The only update: don't bother with a faer-backed
intermediate version of milestone 6 — go straight to a pulp ×4 inner
kernel with fused threshold check. The Step 0b spike already proved
the intermediate has no positive value.

## Caveat — emit cost not measured

Threshold=0.5 on random unit-norm d=1536 vectors gives ~0 edges, so
the emit/concat cost wasn't exercised. If a real workload has many
edges, the fused architecture might pull ahead via reduced memory
write traffic on the score-tile that's "going to be discarded
post-threshold anyway." Still: the *kernel-bound* cost is what
dominates, and that's where Plan A's microkernel work pays off.

## Cross-arch sanity (still pending)

Re-run `examples/probe_pulp_dot.rs` on a Linux x86 machine with AVX2
before milestone 4 to confirm pulp ×4 matches `std::arch` AVX2+FMA
there too. This is optional and can ride along on the CI matrix when
milestone 1 sets it up.

# v1.2 Outcome — M5+M7 reverted, simd library kept as foundation

**Date:** 2026-05-01

After M1-M7 landed (microkernel + fused kernel + wired into `ip_batch`
and `similarity_graph`), end-to-end measurements showed v1.2 is
*slower* than v1.1 (faer) at every production shape:

| Operation | v1.2 (fused, post-cache-blocking) | v1.1 (faer) | Delta |
|---|---|---|---|
| `similarity_graph` (n=20k, d=1536) | 714 ms | 656 ms | 1.09× slower |
| `search` top-10 (nq=1000, n=20k, d=1536) | 80 ms | 69 ms | 1.16× slower |
| `range_search` thr=0.5 (same shape) | 77 ms | 64 ms | 1.20× slower |

**Major findings during the build:**

1. Naive `batched_dot_tile` (4×4 register-blocked, no outer cache
   blocking) was 2× slower than faer on search shapes (q_h=32,
   x_w=20000) due to redundant DRAM reads of x. Adding an outer
   X-cache block (`CACHE_X_BLOCK=64`) closed this gap to ~17%.
2. The remaining ~15-20% gap appears inherent — faer's mature GEMM
   has optimizations (software prefetching, packed kernels, more
   sophisticated parallelism strategy) that we haven't replicated.
3. The fused threshold-emit kernel didn't deliver the expected
   `similarity_graph` speedup either, for reasons Step 0b already
   flagged: the score tile fits L2 on M-series, sparse thresholds
   make the v1.1 scan cheap, and the inner-kernel speed is the
   dominant factor.

**Decision: revert M5 (ip_batch wiring) and M7 (similarity_graph
wiring).** The `rustcluster` library stays on the v1.1 hot path
(faer + scalar GEMV). The `rustcluster-simd` workspace crate stays
as a fully-tested library — `dot_f32`, `l2sq_f32`, `batched_dot_tile`,
`fused_threshold_emit_tile_ip/_l2` are all correct and benchmarked,
ready to swap in once the kernel-quality gap is closed.

**Future work (post-v1.2):**
- Software prefetching in the 4×4 microkernel
- BLIS-style packed kernels (rearrange A and B blocks for sequential
  access)
- Larger register tiles (4×8 or 8×4) on Apple Silicon — needs careful
  register-pressure analysis
- Per-arch hand-tuning where pulp leaves performance on the table
- AVX2 / AVX-512 cross-validation on x86 (CI runner)

The `rustcluster-simd` library is the foundation for that work; M1-M6
isn't wasted, just not yet on the production hot path.
