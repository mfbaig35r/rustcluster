implementing pure-Rust SIMD distance kernels and a fused threshold-emit

property** — no BLAS, no SimSIMD C dependency. The work is scoped at
~7-11 engineer-weeks if the validation prototype (Step 0 below) confirms
the approach.

## What v1.0 and v1.1 already built

The library has flat exact indexes built around faer's matmul. Bench at
n=20k, d=1536, threshold=0.5 (Apple Silicon M-series):

- `similarity_graph` at d=128: **2.3× faster** than FAISS (the production
  workload sweet spot).
- `similarity_graph` at d=1536: **0.77× of FAISS** (the gap this plan
  closes).
- `search` and `range_search` at d=1536: 0.32-0.37× of FAISS.

Phase-1 profiling (committed under `examples/profile_*.rs`) showed
**99.9% of `ip_batch` time at d=1536 is faer's SIMD microkernel itself**
— there's no overhead at the call site to optimize. Faer's pure-Rust
GEMM lags hand-tuned BLAS by 20-30% for K-dominated shapes, which is
exactly what raw embedding dimensions like `text-embedding-3-small`
(1536d) produce.

See `docs/index-faer-tuning-plan.md` for the full diagnostic.

## What we evaluated and rejected

Three alternatives were researched in depth (read in this order if you
want the full reasoning):

1. **Bundle BLAS per platform** (`docs/blas-backend-research-prompt.md` +
   `docs/blas-backend-deep-research.md`). Verdict: **no.** Closes the
   gap (~1.0× FAISS) but ABI clash risk with NumPy's bundled OpenBLAS,
   30MB wheel growth, manylinux pain, and 50-100 hours/year maintenance
   for a solo project. Cannot beat FAISS — only match. Cannot do
   fused kernels because BLAS is opaque.

2. **Wrap SimSIMD via the `simsimd` crate**
   (`docs/simsimd-rust-research-prompt.md` +
   `docs/simsimd-overview-deep-research.md`). Verdict: **no for v1.2.**
   1-2 weeks to integrate, immediate access to SimSIMD's well-tuned C
   kernels, but introduces a C build dependency (breaks pure-Rust
   property) and the f32 dot at d=1536 on aarch64 is actually 0.83× of
   OpenBLAS — i.e., simsimd's f32 perf is comparable to BLAS, not
   dramatically better. The "could beat FAISS" property comes from the
   fused threshold kernel, which we'd write ourselves regardless of
   whether the inner dot uses simsimd or pulp.

3. **Apple AMX direct access.** Verdict: **defer to v1.3+.** ~3×
   speedup on M-series at d≥512, but the `amx-rs` crate carries a
   "DON'T USE IN PRODUCTION" warning, requires nightly + inline asm,
   and the ISA is officially undocumented. Wrong risk profile for a
   published library.

## What we're building (v1.2)

**Path B: pure-Rust SIMD kernels using `pulp`.**

`pulp` is the SIMD abstraction crate that faer itself uses. It provides
runtime dispatch across AVX2, AVX-512, and NEON from a single kernel
implementation. Faer's matmul — the kernel we currently benchmark
against FAISS — is built on pulp. We're not introducing a research-
grade abstraction; we're using the same proven tool, at the same level
of abstraction, that already underpins our existing implementation.

Concretely, four operations:

1. **f32 dot product** (variable d) — single-vector microkernel.
2. **f32 squared L2 distance** (variable d) — derived from dot via the
   identity ‖q-x‖² = ‖q‖² + ‖x‖² - 2·q·x.
3. **Tile-shaped batched dot** (matrix microkernel) — replaces the inner
   `faer::matmul` call inside `ip_batch`.
4. **Fused dot + threshold + emit** — the `similarity_graph` inner
   kernel that never materializes the score matrix. **This is where
   the "could beat FAISS" property lives.**

Cosine = normalized dot, no separate kernel. We are f32-only (matches
v1.0's design choice). No exotic types (BF16, int8, Hamming, KL
divergence) — we don't need them and the perf comparison won't be
affected.

## Why pulp over raw `std::arch`

Two parallel deep-research outputs evaluated this. The earlier one
recommended raw `std::arch` intrinsics with three separate per-arch
kernels (AVX2, AVX-512, NEON), estimating ~20-24 engineer-weeks. That
estimate is correct for raw intrinsics.

The case for pulp:

- One kernel runs on AVX2, AVX-512, and NEON via runtime dispatch.
  You write the inner loop using `pulp::WithSimd`; pulp generates the
  per-architecture variants and selects at startup.
- No `unsafe` intrinsics, no `#[target_feature]` minefield.
- Faer's matmul achieves 70-80% of OpenBLAS performance using pulp.
  That's enough to close most of our gap — the architectural advantage
  of fused kernels covers the rest.

The cost: roughly 10-20% peak performance vs hand-tuned intrinsics for
the innermost microkernel. For our v1.2 target this is acceptable
because (a) we beat FAISS via the fused kernel anyway, not via
microkernel speed, and (b) hot-path tuning with raw intrinsics can
land in v1.3 if measurements show it's needed.

If pulp turns out to be insufficient for the inner kernel — Step 0
below validates this — we fall back to raw `std::arch` for AVX-512 and
NEON, accepting the ~2× effort increase.

## Scope: in and out

**In scope for v1.2:**

- pulp-based f32 dot, L2, batched-dot kernels in a new module
  (`src/index/simd/`).
- Fused dot + threshold + emit kernel for `similarity_graph`.
- Replacement of `faer::matmul` calls inside `src/index/kernel.rs`'s
  `ip_batch()` with the new microkernels.
- Per-architecture tuning for AVX2, AVX-512, and NEON via pulp's
  dispatch.
- Property-based correctness tests vs scalar reference.
- Bench against the v1.1 baseline at all current shapes
  (`benches/index.rs`).

**Out of scope for v1.2 (deferred):**

- Apple AMX direct access — risky, undocumented, alpha crate.
- ARM SVE/SVE2 specialization — only Graviton3+ has it, hard to test
  in CI, low user base.
- BF16, int8, complex types, exotic distance functions.
- New public API. The pyclass surface (`IndexFlatL2`, `IndexFlatIP`,
  their methods) does not change. Callers see only better performance.

## Step 1+: implementation (~7-11 weeks if Step 0 validates pulp)

### Module structure

Add a new module `src/index/simd/`:

```
src/index/simd/
  mod.rs                  # public API (dot_f32, l2sq_f32, batched_dot, fused_threshold_emit)
  scalar.rs               # scalar reference implementations (also fallback)
  pulp_kernels.rs         # pulp-based primary implementations
  pulp_fused.rs           # the fused similarity_graph kernel
  tests.rs                # property tests vs scalar
```

Update `src/index/mod.rs` to declare the new submodule. Keep the
existing `kernel.rs`, `flat.rs`, `similarity_graph.rs`,
`persistence.rs`, etc. — they call into `simd::` instead of
`faer::matmul`.

### Public Rust API of `src/index/simd/`

```rust
// Scalar f32 dot — for tests and small fallback.
pub fn dot_f32_scalar(a: &[f32], b: &[f32]) -> f32;

// SIMD f32 dot product (pulp-dispatched at runtime).
pub fn dot_f32(a: &[f32], b: &[f32]) -> f32;

// SIMD f32 squared L2 distance.
pub fn l2sq_f32(a: &[f32], b: &[f32]) -> f32;

// Tile-shaped batched dot: `out = q (h, d) @ x (w, d)^T`, shape (h, w).
// Replaces the faer::matmul call in ip_batch for tiles.
pub fn batched_dot_tile(
    q: &[f32], q_h: usize,
    x: &[f32], x_w: usize,
    d: usize,
    out: &mut [f32],
);

// Fused similarity-graph kernel: compute upper-triangle pairs of inner
// products and emit (i, j, score) when score >= threshold. Never
// materializes the (h × w) score tile.
pub fn fused_threshold_emit_tile(
    x: &[f32], h: usize, w: usize, d: usize,
    threshold: f32,
    edges: &mut Vec<(u32, u32, f32)>,  // local emit buffer
);
```

The exact signatures may shift — `pulp::WithSimd` has its own ergonomic
constraints — but the surface is small and the four functions above are
the public contract.

### Wiring into existing code

`src/index/kernel.rs::ip_batch` currently calls `faer::matmul`. It
becomes a thin orchestrator that calls `simd::batched_dot_tile`.

`src/index/similarity_graph.rs` currently runs `ip_batch` per tile,
materializes the score tile, and runs a separate threshold loop. The
new path uses `simd::fused_threshold_emit_tile` per tile and skips
score-matrix materialization entirely. **This is the architectural
change that beats FAISS, not the inner microkernel speed.**

`Cargo.toml` adds `pulp = "0.x"` as a direct dependency (faer already
uses it transitively, so this is making an existing transitive dep
direct).

The existing public API (`IndexFlatL2`, `IndexFlatIP`, methods, persistence
format) does not change. All existing tests must continue to pass.

### Performance targets

After v1.2 lands (n=20k, d=1536, threshold=0.5, Apple Silicon):

- `similarity_graph` at d=1536: **at least 1.0× of FAISS** (currently
  0.77×). The fused kernel is the lever.
- `similarity_graph` at d=128: **stay at or above 2.0× faster** than
  FAISS (currently 2.32×). Don't regress the d=128 case while chasing
  d=1536.
- `search` and `range_search`: target within 2× of FAISS at d=1536
  (currently 0.32-0.37×). The fused kernel doesn't directly help these,
  but the better microkernel does.

The minimum acceptable outcome is "no regression at d=128, d=1536
≥ 0.95× of FAISS." Anything better is a bonus. Anything worse means
the implementation is wrong somewhere.

## Testing strategy

### Property-based tests (per kernel)

For each of `dot_f32`, `l2sq_f32`, `batched_dot_tile`, generate random
vectors of varied length and compare SIMD output against
`dot_f32_scalar` reference within float tolerance:

```rust
proptest! {
    #[test]
    fn pulp_dot_matches_scalar(d in 1usize..2000, seed: u64) {
        let mut rng = StdRng::seed_from_u64(seed);
        let a: Vec<f32> = (0..d).map(|_| rng.gen_range(-1.0..1.0)).collect();
        let b: Vec<f32> = (0..d).map(|_| rng.gen_range(-1.0..1.0)).collect();
        let scalar = dot_f32_scalar(&a, &b);
        let simd = dot_f32(&a, &b);
        assert_relative_eq!(scalar, simd, max_relative = 1e-5);
    }
}
```

### Integration tests (via existing pytest)

The existing `tests/test_index_flat.py`, `tests/test_index_similarity_graph.py`,
and `tests/test_index_persistence.py` exercise the full Python API.
After milestone 4 and milestone 6, all must stay green. **They are
end-to-end correctness tests for the new kernels** — if a kernel
miscomputes, brute-force-parity tests in `test_index_similarity_graph.py`
will catch it.

### Per-architecture CI

Add an aarch64 runner to GitHub Actions if not already present.
GitHub now provides `ubuntu-22.04-arm64` runners. The CI matrix:

- Linux x86_64 (default, has AVX2)
- Linux aarch64 (NEON)
- macOS arm64 (NEON)
- macOS x86_64 (older Intel Macs, AVX2)
- Windows x86_64 (AVX2)

Skip AVX-512 in CI unless GitHub provides AVX-512-capable runners — most
don't. Trust the runtime dispatch and test on a real AVX-512 machine
manually if available.

### Performance regression tests

`tests/test_index_perf.py` already has 6 perf-marker tests gated by
`pytest -m perf`. Update the assertions after v1.2 lands:

- `test_search_within_4x_of_faiss` → `within_2x_of_faiss`
- `test_range_search_within_3x_of_faiss` → `within_2x_of_faiss`
- `test_similarity_graph_within_2x_at_d1536` → `within_1.2x_at_d1536`
  (or beats FAISS outright)

These are the gates that "v1.2 closed the gap" must clear.

## Risks and mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| pulp underperforms hand-tuned intrinsics by >30% at the inner microkernel | medium | Step 0 validates this in 1-2 days. If it fails, fall back to raw `std::arch` per the second deep-research plan (~20 weeks). |
| Subtle SIMD bugs (mask handling, tail loops, NaN propagation) | high | Property-based tests vs scalar reference, exercised in CI. The existing pytest brute-force-parity tests catch end-to-end miscomputes. |
| Per-arch CI (especially aarch64) introduces new failure modes | medium | Add aarch64 runner before milestone 1; validate the CI shape before doing real kernel work. |
| Fused kernel has an off-by-one bug at tile boundaries | high | Property test that `fused_threshold_emit_tile(...)` produces the same edge set (after sorting) as `batched_dot_tile + threshold filter`. |
| Effort estimate is too aggressive | medium-high | The 7-11 week range already absorbs most slippage. If milestone 5 (fused kernel) pushes 4+ weeks, escalate — the architecture might not be right. |
| pulp's API changes between versions | low | Pin to a specific minor version. Faer pins it tightly too. |
